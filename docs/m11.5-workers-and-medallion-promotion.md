# M11.5 — Workers + medallion promotion

**Status:** pending

**Source:** [medallion-and-workers.md](medallion-and-workers.md) §1.3 + §3

## Goal

Stand up the ARQ + Redis worker queue, plus the silver and gold layers of the medallion that the workers operate on. M6.5 produces bronze writes; this milestone consumes them, normalizes into silver, recomputes hot + cold gold tables, compacts JSONL → Parquet, and enforces retention. Replays, classifications, and promotions become ARQ jobs (with cron schedules) so M12's FastAPI control plane can enqueue rather than spawn `asyncio.create_task`.

## Behavior

### `replay_run` job

- Loads `Recording` from silver via `RecordingRepository`.
- Spawns its own `BrowserSession` (no sharing across jobs — Playwright contexts are not concurrency-safe).
- Wires the `Executor` event-emitter callback (M12 hook) to `publish_progress(redis, run_id, ...)`.
- Runs `executor.run()`; persists `RunResult` via `RunResultRepository`.
- Enqueues `generate_run_summary(run_id=...)` as a follow-up job.
- Enqueues `promote_silver_to_gold(recording_id=...)` so the dashboard reflects the new run.
- On uncaught exception, publishes `{"type": "run_failed", ...}` and re-raises so ARQ records the failure.

### `classify_recording` job

- Loads recording's actions; for each action, runs `HeuristicClassifier`; if `confidence < threshold`, awaits `LLMClassifier.classify(...)`.
- Updates `RecordedActionRow.semantic_intent`, `classification_confidence`, `classification_reasoning`.
- Each LLM call writes bronze via `BronzeWriter.write_llm_call(...)` (M9 contract).
- One row update per action — no transaction for the whole recording (matches the M5 caller-owns-transaction pattern).

### `promote_bronze_to_silver` job

- Idempotent. Checks `RecordedActionRow` count for the recording; if equal to JSONL line count, returns 0.
- Reads `bronze_store.get(paths.recording_events_jsonl(recording_id))`, splits by `\n`, validates each line via Pydantic.
- Inserts `RecordedActionRow` + `NetworkEventRow` rows. Wraps in a single session.
- Logs malformed envelopes via `structlog.warning(...)` and skips them — bronze keeps the original raw record.

### `compact_bronze_to_parquet` cron job (every 15 min)

- Walks `bronze_store.list("recordings/")`; for each recording with a `raw_events.jsonl` and either no `raw_events.parquet` or one older than the JSONL, calls `compact_recording_jsonl_to_parquet(...)`.
- pyarrow reads the JSONL with a stable schema (lifted from `RecordedAction` via `pydantic_to_arrow_schema(...)`). Writes Parquet with snappy compression.
- Registers a new `bronze_artifacts` row with `kind="event_parquet"`.

### `promote_silver_to_gold` job (cron hourly + on-demand)

- Calls `recompute_gold_hot(session, recording_id=None)` then `ColdGold(...).recompute_*` for each cold table.
- Idempotent: hot tables use SQLAlchemy `merge` / `upsert`; cold parquet files are atomically replaced (write to `<file>.tmp`, then rename).
- When `recording_id` is supplied, only that recording's metrics + replay script are recomputed (used by M11 confirmation flow and M12 on-demand endpoint).

### `prune_old_artifacts` cron job (daily at 03:00)

- Reads retention config from `Config` (per-`kind` retention days).
- For each `BronzeArtifactRow` older than its kind's window: `bronze_store.delete(row.path)`, then delete the row.
- `dry_run=True` reports what would be deleted without acting.

### Redis pub/sub for live progress

- `publish_progress(redis, run_id, event)` calls `await redis.publish(f"run:{run_id}", json.dumps(event))`.
- `subscribe_run(redis, run_id)` yields events from `pubsub.listen()`. Used by the M12 WebSocket bridge.
- Events follow the [data-capture.md §7](.claude/plans/data-capture.md) taxonomy: `run_started`, `action_started`, `action_succeeded`, `action_failed`, `recovery_started`, `recovery_succeeded`, `recovery_failed`, `run_finished`.

### Concurrency tuning

- **Single Redis instance.** Docker `redis:7-alpine` for local dev. Per-worker connection pooling via `redis.asyncio.ConnectionPool(max_connections=20)` shared across all jobs in the same process.
- **Two queues with separate concurrency limits.**
  - `replay_queue` — `max_jobs=2` per worker (browsers are heavyweight, ~250 MB each). FastAPI enqueues `replay_run` here.
  - `medallion_queue` — `max_jobs=10` per worker (IO-bound: file reads, SQL, Parquet). Cron jobs and silver/gold promotion run here.
  - Spawn one ARQ worker per queue: `arq rpa_recorder.workers.settings.ReplayWorker` and `arq rpa_recorder.workers.settings.MedallionWorker`. Both inherit shared `WorkerSettings` plumbing but override `queue_name` and `max_jobs`.
- **`max_tries=1` on `replay_run`.** Replays are not idempotent — a partial run leaves attempt rows and bronze artifacts that a retry would clobber. ARQ's default 5 retries is wrong for this job. All other jobs (silver/gold promotion, compaction, prune) keep `max_tries=3` since they're idempotent.
- **Job timeouts.** `replay_run.job_timeout=1800` (30 min). `prune_old_artifacts.job_timeout=1800` (can grow with bronze size). All other jobs default to 300 (5 min).
- **Distributed lock for gold promotion.** Two workers running `promote_silver_to_gold` simultaneously would race on cold-Parquet writes. Use Redis `SET NX EX` as a leader lock:
  ```python
  lock_acquired = await redis.set("lock:gold_promote", worker_id, nx=True, ex=600)
  if not lock_acquired:
      logger.info("gold_promote already running on another worker; skipping")
      return
  try:
      await recompute_gold_hot(session)
      await ColdGold(root).recompute_classifier_accuracy(session)
      ...
  finally:
      await redis.delete("lock:gold_promote")
  ```
- **Browser-launch semaphore.** Each `ReplayWorker` process additionally caps in-flight browser launches with `asyncio.Semaphore(max_jobs)`. Belt-and-suspenders against accidental over-parallelism if `max_jobs` is misconfigured.
- **Sync libraries wrapped in `asyncio.to_thread`.** DuckDB SQL execution and pyarrow Parquet writes are CPU/sync; wrap every call site so the event loop isn't blocked:
  ```python
  table = await asyncio.to_thread(lambda: duckdb.sql(query).arrow())
  ```
- **Structured concurrency for fan-out.** Cold-gold recomputation runs four independent table builds. Use `asyncio.TaskGroup` (Python 3.11+) so a failure in one cancels siblings cleanly:
  ```python
  async with asyncio.TaskGroup() as tg:
      tg.create_task(cold.recompute_classifier_accuracy(session))
      tg.create_task(cold.recompute_llm_costs_daily(session))
      tg.create_task(cold.recompute_training_data(session))
      tg.create_task(cold.recompute_replay_scripts(session, recording_id))
  ```
  Wrap each task body's logic in try/except to convert per-table failures into structured logs without poisoning siblings — idempotency means the next hourly run retries.
- **Graceful shutdown.** ARQ handles SIGTERM by stopping new dequeues and draining in-flight jobs. Configure `WorkerSettings.shutdown_timeout=60` so the worker waits up to 60 s before forcing exit. `replay_run` jobs check a cancellation flag inside the per-action loop to abort cleanly mid-replay (persists `RunResult.status=CANCELLED`).
- **FastAPI backpressure.** Before enqueuing, check queue depth via `await pool.queue_size()`. If above `Config.max_queue_depth` (default 100), return HTTP 429. Implementation lives in M12; M11.5 ensures `pool.queue_size()` is exposed.
- **DB sessions per job.** Each job opens its own `AsyncSession` via `get_session()`. The `AsyncEngine` is a process-global created in `WorkerSettings.on_startup` and torn down in `on_shutdown`. Sessions are never reused across jobs.
- **Browser sessions per job.** Each `replay_run` invocation creates and tears down its own `BrowserSession`. Sharing across jobs deadlocks Playwright; not concurrency-safe.
- **SQLite write contention.** SQLite has a single writer; with multiple workers writing concurrently (silver inserts, gold upserts, attempt rows), `aiosqlite` waits for the lock — adds tail latency under load. Mitigations: `PRAGMA journal_mode=WAL` (set in `init_db()` from M5), `PRAGMA busy_timeout=5000`, and recommend Postgres for >2 concurrent writers. Document this gating threshold in `pyproject.toml`'s extras: `[postgres]` is no longer optional past that scale.
- **Per-job logging context.** Each job's entry calls `structlog.contextvars.bind_contextvars(job_id=ctx["job_id"], queue=ctx["queue_name"])`. ARQ's default logging is poor; this makes structured logs filterable per run.

## Known pitfalls

- **`arq` v0.25 + asyncio v3.14.** ARQ uses `asyncio.run` internally. Run workers via `arq rpa_recorder.workers.settings.WorkerSettings`, not `python -m`. Don't try to embed in the FastAPI `lifespan` — separate process.
- **DuckDB threads + asyncio.** `duckdb.sql()` is synchronous and CPU-bound; wrap in `await asyncio.to_thread(lambda: ...)` inside async code so the event loop isn't blocked. The pattern is fine for our scale (single-host) but worth noting.
- **PyArrow schema drift.** If `RecordedAction` schema changes (adds a field), old Parquet files become structurally invalid for unioned scans. Add a `schema_version` column to the parquet and bump it on changes; `ColdGold.query()` filters by version.
- **Redis pub/sub does not buffer.** Subscribers connecting after a `replay_run` started miss the early events. M12 should also write events to a per-run list (`LPUSH events:<run_id>`) so late subscribers can replay history. Trade-off: extra Redis writes.
- **Gold recompute can race with silver inserts.** Hot-gold recompute reads silver in a transaction; silver inserts use a separate session. Use `READ COMMITTED` isolation (SQLite default) so gold sees a consistent snapshot.
- **Idempotency of `promote_silver_to_gold`.** Hot tables use upsert on PK. Cold parquet files are replaced atomically — but a partial replace mid-crash leaves a `<file>.tmp` that the next run cleans up before writing.
- **Cron drift on Windows.** ARQ's cron uses the worker process's local clock. If the worker is paused (laptop sleep), cron jobs fire late. Acceptable for a portfolio demo; production would use APScheduler with a database store.
- **`ConsoleMessage` in executor event hooks.** When M12 wires an event-emitter callback into Executor, the callback signature must accept `dict[str, Any]` only — don't leak Playwright objects through. Workers run in a separate process from Redis subscribers and Playwright objects don't pickle.
- **Backpressure on FastAPI enqueue.** Without a queue-depth check, a misbehaving client can flood Redis. M12 must implement the 429 response described in *Concurrency tuning*; M11.5 exposes `pool.queue_size()` for the check.
- **`asyncio.TaskGroup` and partial failures.** When one task in a group raises, the group cancels all siblings *and* re-raises an `ExceptionGroup`. Cold-gold recompute must catch `ExceptionGroup` at the outer boundary and log per-table failure individually so a single broken table doesn't poison the whole run. Idempotency saves us — the next hourly run retries.
- **Redis lock fairness.** The simple `SET NX EX` lock is non-fair: if worker A holds it for 600 s, worker B that arrived 1 s later loses to worker C that arrives at second 599. For our cron cadence (hourly recompute lasting <1 min) this never matters; for finer-grained jobs use `redlock` instead.
- **Cancellation propagation.** `replay_run` checks a cancellation flag at every action boundary. If FastAPI cancels the job (user clicks "stop replay" in M12 UI), the action loop exits cleanly, persists `RunResult.status=CANCELLED`, and tears down the browser. Don't rely on raw `asyncio.CancelledError` propagation alone — ARQ wraps the worker loop and uncaught cancellation looks like a crash.
- **Semaphore + worker-process boundary.** Each worker process has its own event loop and its own browser-launch semaphore. Cross-process coordination (e.g., capping global browser count to 4 across two workers running 2 each) is not provided. If you need a global cap, use a Redis-backed semaphore (`aioredlock` or a dedicated counter key).
- **`ConnectionPool` leak on worker restart.** ARQ's `on_shutdown` must explicitly `await pool.disconnect()` to release the Redis connections; otherwise a fast restart loop accumulates open file descriptors. Add this to `WorkerSettings.on_shutdown`.
